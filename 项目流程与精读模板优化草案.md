# 项目流程与精读模板优化草案

这份文档整理本轮 `$grill-me` 后已经锁定的流程和模板决策。当前只确定框架与规则，不批量迁移旧笔记，不改已有文献内容。

## 当前目标

优化研究阅读库的两个部分：

- **收文献流程**：不只服务 arXiv，也要能处理更广泛来源的完整文献。
- **精读稿模板**：保留完整论文讲解的内容密度，但降低阅读时的认知成本，让读者更快知道作者到底想做什么。

## 已锁定原则

- **PDF 是统一入口，不等于元数据解析优先级。** 本地 PDF、下载得到的 PDF、以及其他能拿到完整文献的渠道都应能进入流程；但如果识别出 arXiv、DOI 或正式发表信息，应优先用这些来源确认文献身份。
- arXiv 仍然优先支持：arXiv 元数据、source package、PDF 抽图仍是最可靠的自动化路径之一。
- 非 arXiv 文献没有 source package 时，可以接受从 PDF 尽量抽图或截图，不强求完整源文件抽图。
- 精读稿面向“正在精读的我”，第一目标是快速掌握作者具体在做什么、为什么要做、用什么证据支撑。
- 顶部必须保留 **一句话总结**，但允许变成两句话；它要承担“作者行动定位”的功能。
- 顶部新增 **论证链**，默认是压缩箭头链；复杂论文允许拆成两行，但仍保持箭头链。
- 顶部用 **主张-证据速览** 替代 **贡献速览**。
- 不需要在顶部加入“回看理由”。
- 现阶段只改框架和后续模板/脚本设计，不批量迁移旧笔记。

## 核心诊断

现在的精读稿不是信息不够，而是前部逻辑总结不够锋利：

```text
论文在做什么
-> 作者为什么认为这个问题重要
-> 现有方法缺在哪里
-> 作者提出什么主张/方法
-> 用什么证据支撑
-> 结论成立到什么边界
```

后续优化应避免把精读稿进一步拉长，而是把“完整内容”重新组织成更容易扫读的层次。

## 精读稿顶部框架

目标：读者在一屏内抓住论文论证路线。

```markdown
## 一句话总结

本文针对 [对象/问题]，通过 [核心动作：反驳/证明/构造/提出/测量/解释/综述]，产出/支持 [方法/模型/框架/系统/结论]。
作者这样做是因为 [现有缺口/争议/实践困难] 使得 [关键问题] 还没有被充分解决。

## 论证链

研究对象/问题 -> 现有缺口 -> 核心主张/方法 -> 机制解释 -> 关键证据 -> 适用边界

## 主张-证据速览

- 贡献 1 实际主张：……
  证据位置：Figure/Table/Experiment/Proof/Case ……
  边界：仅在证据明显不足或有外推风险时补一句。
- 贡献 2 实际主张：……
  证据位置：……

## 章节导航

- 1 Introduction：提出 [研究对象/问题] 和 [现有缺口]，为后文的 [核心主张/方法] 铺垫。
- 3 Method：承接 [现有缺口]，展开 [机制解释]。
- 4 Experiments：检验 [关键证据] 是否足以支撑 [核心主张]。
```

### 顶部规则

- `一句话总结` 默认两句话：第一句回答作者到底做什么，第二句回答作者为什么要做。
- `论证链` 默认单行；复杂论文允许两行，但不得降级成空泛标签。
- `论证链` 每个节点必须写具体内容，不能只写“现有缺口 / 关键证据 / 适用边界”这类栏目名。
- `主张-证据速览` 按作者声称的贡献逐条改写为可检验主张，不照抄 contribution。
- `主张-证据速览` 每条必须给出证据位置；只有外推风险或证据不足明显时才写边界。
- `章节导航` 只服务跳读，每行写“章节功能 + 与论证链的关系”，不再承担长摘要功能。

## 正文结构

正文仍然是完整论文讲解，不是短摘要；但默认结构要比旧模板更轻。

### Section 处理规则

- 保留原文一级 section。
- subsection 可以合并讲解；只有在以下情况单独列标题：
  - 承担独立论证功能，例如定义问题、提出关键假设、证明关键命题。
  - 包含关键方法、实验、公式或图表，不单列会导致证据位置不清。
  - 容易误读，尤其涉及概念区分、反驳对象、限制条件。
- 不按篇幅机械拆 subsection；长不等于重要。

```markdown
### Section N. 原文标题

这一节自然讲解作者在这里推进了什么问题、如何承接前文、为后文提供什么基础。不要强制写 `本节作用` 标签，也不要求固定句式。

**关键证据 / 图表 / 公式**：
- Figure 1：说明……
- Table 2：显示……
- Equation (3)：定义……

其中最关键的 1-2 个图表或公式需要展开解释：它支撑哪条主张、证据含义是什么、读者应注意什么。
```

### 图表与公式规则

- 图表尽量全收，公式只收关键公式。
- 非关键公式可以在正文讲解中概述用途，不必逐条展开变量含义。
- 图表嵌入分层：
  - **必嵌入**：核心方法图、主实验结果图、关键对照表。
  - **可文字说明**：流程辅助图、重复性附录图、只展示样例的图。
  - **只进图片索引**：不影响正文理解的补充图。
- `关键证据 / 图表 / 公式` 默认用混合写法：先用短列表帮助扫读，再对最关键的 1-2 个展开解释。
- 只有关键图表/公式需要明确写出“它支撑哪条主张”；普通辅助图表只解释内容和用途。

### 加重工具触发规则

旧模板中的 `高层故事流` 和 `论证功能表` 不再作为每节默认栏目，只在复杂章节中临时使用。触发条件：

- **机制复杂**：方法/理论章节里模块很多、公式推导多、步骤链长。
- **证据复杂**：多个实验、表格、图互相支撑不同主张。
- **论证复杂**：作者在反驳、区分概念、修正已有观点，容易读偏。

章节很多本身不触发加重工具。

## 文末结构

取消旧模板中的：

- `主张-证据-边界矩阵`
- `精读路线 / 为什么需要回看`

保留并强化 `局限与可追问点`，由它承担边界复盘：

```markdown
## 局限与可追问点

### 作者承认的局限

- …

### 证据不能推出的结论

- …

### 可继续追问的问题

- …
```

## 精读稿模板草案

后续生成 `<stem>_精读稿.md` 时，默认使用以下结构：

```markdown
# 《论文标题》精读稿

## 一句话总结

本文针对 [对象/问题]，通过 [核心动作]，产出/支持 [方法/模型/框架/系统/结论]。
作者这样做是因为 [现有缺口/争议/实践困难] 使得 [关键问题] 还没有被充分解决。

## 论证链

[具体研究对象/问题] -> [具体缺口] -> [作者的具体主张/方法] -> [机制关键词] -> [证据类型+关键结果] -> [适用边界]

## 主张-证据速览

- 贡献 1 实际主张：……
  证据位置：……
  边界：……
- 贡献 2 实际主张：……
  证据位置：……

## 章节导航

- 1 Introduction：……
- 2 Related Work：……
- 3 Method：……
- 4 Experiments：……
- 5 Discussion / Conclusion：……

## 1 原文 Section 标题

自然讲解本节如何推进全文论证、作者在这里说明了什么、与前后文的关系是什么。
```

## 收文献流程

目标：以 PDF 为中心，兼容 arXiv 之外的完整文献来源。

```text
获得 PDF
-> 识别来源与文献身份
-> 查重
-> 建立/补全文献笔记
-> 记录 pdf / reading / images / image_index 等字段
-> 抽取图片并建立图片 index
-> 生成精读稿
-> 更新 reading 字段与 checklist
-> 运行地图检查脚本
```

### 元数据识别顺序

```text
识别来源
-> 若是 arXiv：优先使用 arXiv 元数据 / source package / PDF
-> 若 PDF 中可识别 DOI/arXiv/title：用 DOI/arXiv/OpenAlex 反查确认
-> 若无法确认：从 PDF 抽标题、作者、摘要、正文、图表，低置信入库
-> 人工兜底
```

如果同一篇论文同时有 arXiv 版本和正式发表版本，文献笔记默认以正式发表版本为准：

```yaml
doi: 正式发表 DOI
venue: 正式发表 venue
year: 正式发表年份
arxiv: arXiv ID / URL
pdf: 当前实际阅读的 PDF
```

如果 arXiv 预印本和正式发表版标题、作者或内容差异明显，脚本只提示差异，不自动拆分或合并，由人工判断。

### 元数据质量字段

文献笔记中增加或保留两个字段：

```yaml
metadata_source: pdf | doi | openalex | manual | arxiv | webpage | semantic_scholar
metadata_confidence: high | medium | low
```

`metadata_confidence` 按综合标准判定：来源可靠性 + 关键字段是否互相一致。

- `high`：标题、作者、年份、DOI/arXiv/venue 等关键字段来自可靠来源，且互相一致。
- `medium`：核心字段足够建立笔记，但 DOI/venue/作者顺序等仍有一部分待确认。
- `low`：只能从 PDF 或文件名猜出部分信息，可能需要人工确认。

### citekey 与 PDF 命名

当前库已有 `papers/@*.md` 风格，citekey 统一升级为：

```text
@姓氏年份短题名
```

短题名默认 1-4 个关键词，用连字符连接：

```text
@wang2024large-scale-vbts
@li2025vbts-classification-review
```

规则：

- 初始入库时保留原 PDF 文件名。
- 元数据达到 `high` 或 `medium` 后，可以半自动重命名为 citekey。
- 半自动重命名必须先生成预览，用户确认后再执行。
- 重命名时同步更新文献笔记中的 `pdf / images / image_index` 等路径字段。
- citekey 冲突时脚本只提示冲突，不自动兜底，不自动加序号，要求人工确认最终名。

### 去重规则

新 PDF 入库时，默认按以下顺序查重：

```text
DOI / arXiv ID -> 标题规范化 -> PDF 文件哈希
```

发现可能重复时，不自动阻止，也不自动合并；提示用户选择：

- 合并到已有条目。
- 仍然新建。
- 取消。

### 图片处理规则

arXiv：

- 默认先用本地脚本从 arXiv source package / PDF 中抽取图片。
- 保存到 `papers/images/<pdf-stem>/`。
- 生成 `index.md`。

非 arXiv PDF：

- 尽量从 PDF 自动抽取图片对象。
- 抽取失败也生成图片目录与 `index.md`。
- 正文需要时再按需截图或补图。

图片目录随 PDF/citekey 重命名的规则：

- 若尚未有精读稿引用图片：可同步重命名 images 目录，并更新 `images/image_index`。
- 若已有精读稿引用图片：默认不自动改目录，除非用户确认迁移链接。

## 不改动的约束

- 不联动 Zotero。
- 不生成 Zotero URI。
- `reading` 仍然是判断是否已有精读稿的唯一状态源。
- 图片清单仍然放在 `papers/images/<pdf-stem>/index.md`。
- 精读稿正文不在顶部堆叠图片预览。
- 新增或完成精读稿后，仍需维护文献笔记字段与 `论文地图.md` 的快速索引状态。
